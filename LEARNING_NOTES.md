## Як працює система в цілому

```
[Адмін відкриває сайт] → [Вибирає квіз] → [Запускає гру]
         ↓
[Сервер (MacBook)] ←→ [WebSocket з'єднання] ←→ [Телефони гравців]
         ↓
[Автоматичний таймер] → [Наступне питання] → [Підрахунок балів]
```

## Чому Node.js?
Node.js ідеальний для real-time додатків тому що:
1. **Event-driven** - Не блокує виконання поки чекає на відповідь
2. **JavaScript** - Та сама мова що і на фронтенді
3. **npm ecosystem** - Тисячі готових бібліотек

## Що таке State Machine (Машина Станів)?
Наша гра завжди знаходиться в одному з цих станів:
1. **LOBBY** - Гравці підключаються, гра не почалась
2. **QUESTION** - Показується питання, йде таймер
3. **ANSWER** - Час вийшов або всі відповіли, показуємо правильну відповідь
4. **SCOREBOARD** - Показуємо рейтинг між питаннями
5. **FINISHED** - Квіз завершено, фінальний результат

## Як WebSocket відрізняється від звичайного HTTP?
- **HTTP**: Клієнт питає → Сервер відповідає → З'єднання закривається
- **WebSocket**: З'єднання ПОСТІЙНЕ. Сервер може надсилати дані в будь-який момент без запиту від клієнта

## Що таке SQLite і чому він підходить для цього проєкту?

**SQLite** — це база даних, яка зберігається в одному файлі (`data/sessions.db`). Вона не потребує окремого сервера, як PostgreSQL або MySQL — просто файл на диску.

Аналогія: якщо MySQL — це банк із касирами та черга, то SQLite — це записна книжка у ящику столу. Для одного користувача (адміна закладу) — цього більш ніж достатньо.

**better-sqlite3** — бібліотека Node.js для роботи з SQLite. Особливість: вона **синхронна** (блокує виконання поки йде запит), тоді як більшість Node.js бібліотек асинхронні. Це спрощує код — не потрібні `async/await` або callback-и для кожного запиту до БД.

```javascript
// Синхронна робота з better-sqlite3
const sessions = db.prepare('SELECT * FROM sessions').all();
// Виконується одразу, результат у sessions
```

## Що таке QR-код і як сервер його генерує?

**QR-код** — це двовимірний штрих-код, що кодує текст (у нашому випадку — URL). Телефон сканує зображення і відкриває закодовану адресу.

Сервер генерує QR так:
1. Визначає власну IP-адресу в локальній мережі (`os.networkInterfaces()`)
2. Складає URL: `http://10.0.1.36:8080/?room=AB3C7D`
3. Бібліотека `qrcode` перетворює цей рядок у PNG-зображення
4. Браузер отримує зображення через звичайний HTTP-запит `GET /api/qr/AB3C7D`

Фронтенд просто ставить `<img src="/api/qr/AB3C7D">` — браузер сам завантажує картинку. Жодного JavaScript для QR не потрібно.

## Що таке i18n (інтернаціоналізація)?

**i18n** — скорочення від "internationalization" (між першою і останньою літерою — 18 літер). Означає підтримку кількох мов в інтерфейсі.

У нашому проєкті підхід простий:

```javascript
// Словник рядків для кожної мови
const TRANSLATIONS = {
  uk: { join: '🚀 Приєднатись', correct: 'Правильно!' },
  en: { join: '🚀 Join',        correct: 'Correct!'    }
};

// Хук повертає функцію-перекладач
const [t, lang, setLang] = useLang();
t('join') // → '🚀 Приєднатись' або '🚀 Join' залежно від мови
```

Вибір мови зберігається в `localStorage` — це сховище в браузері, яке зберігається між сесіями (на відміну від `sessionStorage`, яке очищається при закритті вкладки).

## Що таке React hook?

**Hook** — це функція React, яка дозволяє компоненту мати власний стан або підписатись на зміни. Усі хуки починаються з `use`.

`useLang()` — власний хук (custom hook), який:
1. Читає поточну мову з `localStorage`
2. Повертає функцію `t(key)` для перекладу
3. Повертає функцію `setLang(lang)` для зміни мови
4. При зміні мови автоматично перемальовує компонент (через `useState`)

Хуки — спосіб виділити логіку в окремий файл і повторно використовувати її в різних компонентах без дублювання коду.

## Як браузер читає параметри з URL?

URL `http://localhost:8080/?room=AB3C7D` містить **query parameter** `room=AB3C7D`.

```javascript
// Стандартний Web API для читання параметрів URL
const params = new URLSearchParams(window.location.search);
const room = params.get('room'); // → 'AB3C7D'
```

`window.location.search` — це рядок `?room=AB3C7D`.
`URLSearchParams` розбирає його в зручний об'єкт.

Це дозволяє ділитись посиланням виду `/?room=AB3C7D` — гравець відкриває сторінку і поле коду вже заповнене.

## Як пов'язані таблиці в базі даних?

У `sessions.db` три таблиці:

```
sessions       results              question_stats
─────────      ───────────────      ──────────────
id             id                   id
room_code      session_id ──→ id    session_id ──→ id
title          position             question_index
started_at     nickname             total_answered
ended_at       score                answer_0..3
...            ...                  ...
```

`session_id` в `results` і `question_stats` — це **зовнішній ключ (foreign key)**: він посилається на `id` у таблиці `sessions`. Так пов'язуються записи між таблицями.

Щоб отримати всіх гравців сесії №3:
```sql
SELECT * FROM results WHERE session_id = 3 ORDER BY position;
```
